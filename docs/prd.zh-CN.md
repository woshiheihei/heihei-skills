# 产品需求文档 (PRD)：X (Twitter) 深度上下文抓取 Skill

## 1. 文档目的

本文档用于定义 X 单条推文深度上下文抓取 Skill 的产品需求。

本文档回答的是“这个产品要解决什么问题、输出什么结果、如何判断成功”，而不是“内部应该如何实现”。

---

## 2. 产品概述

### 2.1 产品定义

本 Skill 用于围绕一条目标 X 推文，整理出一份高密度、可直接供后续大模型、Agent 或深度研究系统消费的上下文包。

它的目标不是替用户解释内容，而是尽可能完整、准确地整理原始讨论材料，供后续模型自行分析：

* 作者原帖说了什么
* 作者后续回复和补充说了什么
* 评论区里有哪些关键原始观点
* 哪些关键人物参与了讨论
* 每条重要内容对应的作者身份和互动信号是什么
* 讨论依赖了哪些引用推文、文章或外部资料

### 2.2 核心输入

输入为一条目标 X 推文链接。

### 2.3 核心输出

输出为单一最终交付文件，默认保存在当前工作目录下的专用输出目录：

`./x-thread-contexts/`

文件名采用结构化命名规则：

`x-thread-context-<author_handle>-<root_status_id>.md`

例如：

`./x-thread-contexts/x-thread-context-karpathy-2039805659525644595.md`

该文件必须是完整、自洽、机器可直接消费的研究上下文，不依赖用户再打开原始链接或拼接其他中间文件，即可直接上传给 ChatGPT、Gemini 或其他大模型继续做深度研究。

该交付文件的核心价值在于保留原始材料，而不是提供系统解释。

---

## 3. 要解决的问题

在用户当前的研究工作流中，围绕一条高信息密度的 X 推文通常会遇到以下问题：

* 主贴内容与作者后续补充分散在不同位置，理解成本高
* 评论区信息量大，但高价值内容占比低，人工筛选耗时
* 高影响力人物、作者回复、关键争议点和引用资料容易被遗漏
* 后续分析往往缺少结构化、可追溯的上下文材料
* 通用搜索结果往往难以完整覆盖 X 内部实时讨论、对立观点、产品趋势和用户态度
* 深度研究模型即使具备较强分析能力，也常常缺乏足够完整的一手 X 语料作为输入

本 Skill 需要弥补“分析能力强，但 X 一手上下文不足”的缺口，让用户能够先沉淀高质量 X 原始讨论材料，再交由多家深度研究模型继续分析。

---

## 4. 目标用户与使用场景

### 4.1 目标用户

主要面向以下用户：

* 需要把单条推文及其讨论交给大模型继续深度研究的用户
* 需要为后续 Agent 工作流准备高质量输入材料的 AI 工作流使用者
* 需要快速理解一条推文及其争议结构的研究人员或内容工作者

### 4.2 典型使用场景

用户提供一条推文链接，希望系统输出一份结构化上下文，供后续模型直接进行：

* 话题分析
* 观点归纳
* 争议判断
* 舆情研究
* 延伸资料阅读
* 趋势研究
* 用户态度分析

---

## 5. 产品目标

### 5.1 核心目标

Skill 必须能够围绕目标推文产出单一的高质量交付文件，默认保存在 `cwd` 下的 `x-thread-contexts/` 目录内，文件名采用 `x-thread-context-<author_handle>-<root_status_id>.md` 结构，重点保留作者原帖、作者回复、评论区原始内容及其元信息，使后续大模型或 Agent 仅通过消费这一个文件即可直接进入下一步深度分析。

### 5.2 成功标准

若后续大模型仅基于该交付文件，即可较高置信度还原讨论原貌并继续分析，则视为任务成功。至少应满足：

* 能直接读取作者原帖内容
* 能直接读取作者后续回复和补充内容
* 能直接读取评论区关键内容
* 能识别每条重要内容的作者、时间和互动信号
* 能识别哪些外部引用构成了讨论背景

---

## 6. 产品原则与约束

### 6.1 安全与合规优先

本 Skill 的产品形态要求以基于 Chrome 的拟人化浏览和观察为边界，不以程序化接口抓取为目标。

因此，产品层面明确约束如下：

* Agent 应通过 Chrome 浏览器执行浏览任务
* Agent 应模拟人类在 Chrome 中查看、滚动、点击、返回和停留的浏览行为
* 不依赖 X 官方或非官方 API 作为主要抓取路径
* 不以 DOM 解析或网络拦截作为主要信息获取方式
* 不以批量化、非拟人化的高速操作作为产品能力卖点

### 6.2 理解完整性优先于数量完整性

本 Skill 的目标是获得足以支持分析的上下文完整性，而不是追求评论区或引用内容的全量穷举。

### 6.3 作者视角优先

凡属于作者本人主贴、thread、补充说明、对他人的回复或主动引用内容，均应在结果中优先覆盖。

### 6.4 信息密度优先

普通评论不追求数量覆盖，而追求价值覆盖。输出结果应尽量减少噪音和重复。

### 6.5 模型消费优先于人工阅读体验

本 Skill 的输出首先服务于后续大模型和 Agent 的输入质量，而不是优先服务于人工精读体验。

因此，输出设计应优先保证：

* 上下文完整
* 来源关系明确
* 结构稳定
* 便于模型继续抽取、归纳、比较和推理

---

## 7. 范围与非目标

### 7.1 范围内

本 Skill 关注的对象包括：

* 目标 Root Tweet
* 作者发布的后续 thread
* 作者在评论区的关键补充和回复
* 对理解话题有显著价值的评论
* 与核心论点直接相关的引用推文和外部资料

### 7.2 非目标

以下内容不属于本 Skill 的目标：

* 全量抓取整条评论区
* 对所有外部链接做无限深度扩展
* 提供实时监控或持续更新服务
* 追求毫秒级执行效率
* 替代后续分析模型直接给出完整结论

---

## 8. 需求范围与优先级

### 8.1 P0：必须覆盖

以下内容必须出现在最终结果中：

* 目标 Root Tweet 的核心内容
* 作者 thread 的关键内容
* 作者后续补充、澄清或解释
* 作者对其他用户的关键回复

### 8.2 P1：应优先覆盖

以下内容原则上应优先纳入：

* 高影响力人物的评论
* 官方认证账号评论
* 明显会改变讨论权重的行业关键人物评论
* 被作者直接回复的评论
* 引发后续讨论链的重要评论

### 8.3 P2：按价值选择性覆盖

以下内容在具备明显信息增量时应纳入：

* 提供新事实、新证据或新案例的评论
* 提供关键反驳或关键质疑的评论
* 补充历史背景、技术背景或行业背景的评论
* 附带重要来源、截图、链接或数据的评论

### 8.4 P3：通常不纳入

以下内容通常不应成为最终交付重点：

* 纯情绪表达
* 简单附和
* 重复他人观点且无新增信息的评论
* 明显偏离主话题的内容

---

## 9. 功能需求

### 9.1 上下文整合

系统必须能够围绕单条目标推文输出结构化上下文，而不是仅输出主贴文本本身。

### 9.2 作者相关内容优先覆盖

系统必须优先呈现作者本人发布的相关内容，包括 thread、补充说明和关键回复。

### 9.3 原始内容保留

系统必须尽可能保留原始文本内容，不应以系统总结替代原文。

至少应覆盖：

* 作者原帖原文
* 作者后续回复原文
* 评论区关键评论原文

### 9.4 元信息保留

系统必须为重要内容保留必要元信息，至少包括：

* 发布者
* 账号标识
* 时间信息
* 点赞数
* 收藏数
* 转发或其他可见互动信号

### 9.5 高价值评论识别

系统必须能够从评论区中筛选出高价值内容。高价值判断至少应综合考虑以下因素：

* 评论者身份影响力
* 与作者主张的关联程度
* 信息增量
* 传播信号
* 是否成为争议节点
* 是否包含重要来源

### 9.6 引用内容补充

当作者或关键评论明确引用其他内容时，系统应根据来源类型采用不同处理方式：

* 若引用内容仍属于 X 平台内部内容，例如推文、thread 或 X 内文章，则应尽可能完整抓取其原始内容，并嵌入最终交付文件
* 若引用内容属于第三方网站，例如 Anthropic、OpenAI、新闻站点、博客、论文页面或其他官网页面，则以保留原始 URL 为主，不要求在本工具内展开抓取全文

这样处理的原因是：

* X 内部内容通常无法被后续 deep research 模型稳定检索，因此需要在本工具中直接保留
* 第三方公开网页通常可被 ChatGPT、Gemini、Claude 等模型通过自身搜索能力再次访问，因此无需在本工具中重复搬运全文

### 9.7 自主结束判断

系统不要求机械性遍历全部内容，但必须能够在“信息已足以形成认知闭环”时结束任务。

在任务结束时，系统还应执行浏览收尾动作：

* 关闭本次任务过程中为抓取而打开的 X 相关标签页
* 避免让抓取过程中打开的标签页持续堆积
* 若浏览器中还存在用户原本已打开、且与本次任务无关的页面，不应误关闭这些页面

### 9.8 可追溯性

最终结果应具备基本可追溯性，使使用者能够区分：

* 哪些内容来自作者
* 哪些内容来自评论区
* 哪些内容来自引用资料

### 9.9 面向模型的可消费性

最终输出必须适合作为后续大模型或 Agent 的直接输入，至少应满足：

* 单文件可消费
* 章节边界清晰
* 事实、观点、来源关系尽量分离
* 重要上下文不依赖外部补充说明

---

## 10. 输出要求

### 10.1 核心交付物

最终必须输出且只要求用户消费单一交付文件，默认文件名采用：

`x-thread-context-<author_handle>-<root_status_id>.md`

允许系统在内部生成中间材料，但这些内容不应成为用户完成后续研究的前提。对用户而言，该交付文件必须是可直接交付给后续模型的完整文件。

### 10.1.1 输出目录与文件名规则

默认输出目录规则如下：

* 在当前工作目录 `cwd` 下创建或复用 `x-thread-contexts/`
* 最终交付文件默认保存在该目录内
* 不应将大量结果文件直接散落在 `cwd` 根目录中

默认文件名规则如下：

* 前缀固定为 `x-thread-context`
* `author_handle` 使用目标 root tweet 作者的 X handle，去掉 `@`，统一小写
* `root_status_id` 使用目标 root tweet URL 中的 status id
* 最终扩展名固定为 `.md`

示例：

* `./x-thread-contexts/x-thread-context-karpathy-2039805659525644595.md`

若作者 handle 在页面中暂时不可见，可暂用 `unknown-author` 作为占位，但在拿到正确 handle 后应更新为规范文件名。

### 10.2 内容结构要求

该交付文件至少应包含以下信息层次：

1. 目标推文原文与元信息
2. 作者后续回复原文与元信息
3. 关键评论原文与元信息
4. 关键人物内容
5. 关键引用内容

### 10.3 表达要求

输出必须满足以下要求：

* 结构清晰，便于模型稳定解析
* 明确区分作者内容、评论内容和引用内容
* 优先保留原始文本，不以系统主观改写替代事实表达
* 在保证信息密度的前提下控制噪音
* 保证单文件自洽，避免将关键背景散落在其他文件中
* 尽量减少只对人类排版友好但对模型推理无增益的装饰性表达
* 章节标题简短稳定，便于不同模型重复消费
* 不以系统讲解、点评或总结作为主体内容

### 10.4 任务收尾要求

在成功生成交付文件后，系统应进行一次轻量收尾：

* 将最终 Markdown 文件保存到 `cwd/x-thread-contexts/`
* 关闭本次任务中新打开的 X 抓取标签页
* 不保留大量无用中间浏览标签页

---

## 11. 验收标准

### 11.1 结果完整性

验收时应重点判断：

* 是否保留了作者原帖原文
* 是否保留了作者关键回复原文
* 是否保留了评论区关键内容原文
* 是否为重要内容附带了必要元信息
* 是否补充了关键引用内容

### 11.2 结果可用性

该交付文件应达到“可直接上传给后续模型继续分析”的标准，不应要求使用者再次回到原页面补背景，也不应要求使用者再查阅其他中间文件补全上下文。

### 11.3 结果质量

最终交付应尽量满足以下质量特征：

* 高信息密度
* 低重复
* 结构化清晰
* 关键脉络完整
* 来源关系明确
