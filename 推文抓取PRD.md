# 产品需求文档 (PRD)：X (Twitter) 深度视觉抓取 Skill

## 1. 文档目标

本文档是当前正式版本的产品需求文档。

本文档的目标不是重新定义产品方向，而是把以下内容补齐到可以直接指导实现的程度：

* Agent 的自主判断边界
* 抓取范围与优先级
* 状态机和任务生命周期
* 中间产物与目录结构
* 结束条件、降级策略与恢复原则
* 最终 Markdown 交付规范

---

## 2. 产品定义

### 2.1 产品本质

本 Skill 不是传统意义上的爬虫，也不是以全量采集为目标的数据抓取器。

它是一个基于视觉观察和物理交互的研究型 Agent。它像一个非常聪明、非常谨慎的人类研究员一样，在 X 上阅读一条推文及其上下文，并输出一份足以让大模型立即进入深度分析的 Markdown 上下文包。

### 2.2 核心输入

输入为一个目标 X 推文链接。

### 2.3 核心输出

输出为一个任务级目录，其中最重要的交付物是：

* `final.md`：用于后续大模型分析的 Markdown 文件

同时允许并鼓励生成任务过程中的原始资料和中间产物，用于：

* 溯源
* 复盘
* Skill 迭代
* 后续二次分析

### 2.4 核心成功标准

如果一位不熟悉该话题的人类研究员或 ChatGPT Pro 只阅读 `final.md`，就能够快速理解以下问题，则任务视为成功：

* 这条推文在讲什么
* 作者真正想表达什么
* 作者是否在后续补充了关键说明
* 评论区最有价值的支持、反对、质疑和延展观点是什么
* 哪些大 V 或关键人物参与了讨论
* 讨论引用了哪些推文、文章或外部资料
* 整个话题的主要争议点和信息来源是什么

---

## 3. 第一性原则

### 3.1 账号安全高于效率

Agent 必须坚持纯视觉、纯物理交互原则：

* 禁用 API
* 禁用 DOM 解析
* 禁用网络请求拦截
* 禁用 JS 注入和脚本滚动

允许使用的能力仅限于截图、鼠标、键盘、浏览器前进后退、人工节奏等待等拟人化操作。

### 3.2 理解完整性高于机械穷举

Agent 的目标不是把页面滚到底，而是形成对话题的完整理解。

因此，抓取停止条件不以“页面触底”为唯一标准，而以“话题已经基本搞清楚”为主标准。

### 3.3 作者视角绝对优先

凡是作者本人发布、补充、回复、引用的内容，默认优先级最高。

### 3.4 价值密度高于评论数量

普通网友评论不追求全量保留，而追求高价值、高信息密度、高影响力。

---

## 4. 非目标

以下内容不属于本 Skill 的目标：

* 全量抓取整条评论区
* 对所有外部链接做无限深度爬取
* 用程序化接口替代视觉判断
* 追求毫秒级速度
* 构建实时在线服务

---

## 5. Agent 角色定义

### 5.1 Agent 的工作方式

Agent 应被视为一个在 X 上进行调研的信息研究员，而不是规则死板的自动脚本。

它需要基于当前页面内容、上下文关系、平台排序规律和讨论密度，自主判断：

* 哪些评论值得保留
* 哪些引用内容值得下钻
* 是否还需要继续下翻
* 当前信息是否已经足够形成认知闭环

### 5.2 Agent 的判断责任

如果规则与实际页面情况冲突，以 Agent 基于上下文形成的判断为准。

这意味着本 Skill 的设计应显式允许：

* 启发式判断
* 相对排序判断
* 基于身份与内容的综合价值判断
* 在不确定时偏保守地多保留重要内容

---

## 6. 抓取对象与优先级

### 6.1 P0：必须完整抓取

以下内容为绝对核心上下文，原则上必须完整保留：

* 目标 Root Tweet
* 作者发布的完整 Thread
* 作者在评论区的额外补充说明
* 作者对其他人的回复
* 作者主动引用的推文、文章、网页或其他外部内容

### 6.2 P1：必须优先抓取

以下内容通常强烈影响讨论质量和话题判断，应优先保留：

* 高影响力人物的评论
* 官方认证账号评论
* 行业 KOL 或明显知名人物评论
* 高赞、高收藏、高传播信号评论
* 被作者回复的评论
* 引发多人跟进讨论的评论

### 6.3 P2：条件性保留

以下内容可根据上下文判断是否保留：

* 提供新事实、新例子、新证据的评论
* 提供强反驳或关键质疑的评论
* 补充技术背景、行业背景或历史背景的评论
* 附带引用来源、数据、文章或截图的评论

### 6.4 P3：通常忽略

以下内容通常应降权或忽略：

* 纯情绪表达
* 低信息密度附和
* 重复别人观点的评论
* 无新增信息的玩梗或灌水
* 与主话题明显偏离的回复

---

## 7. 高价值评论判断标准

### 7.1 总体原则

不采用固定阈值，而采用综合打分思路。Agent 需要同时观察身份、内容、传播信号和讨论位置。

### 7.2 推荐评估维度

Agent 可使用如下维度进行综合判断：

* `身份权重`
评论者是否为知名人物、认证用户、官方组织、行业 KOL。

* `作者关联度`
该评论是否被作者回复、引用、认同、反驳，或与作者 thread 形成直接互动。

* `信息增量`
该评论是否带来新的事实、案例、证据、解释框架或反驳论点。

* `传播强度`
该评论是否拥有明显较高的点赞、收藏、转发等信号。

* `争议中心性`
该评论是否成为某个关键争议节点，是否吸引多人围绕其继续讨论。

* `来源价值`
该评论是否附带重要链接、引用推文、文章、截图或数据来源。

### 7.3 强制保留规则

当评论满足以下任一条件时，应默认强制保留：

* 评论来自明显高影响力人物或大 V
* 评论者身份本身足以改变讨论权重，例如 Elon Musk 级别人物
* 评论被作者直接回复且内容与核心论点相关
* 评论引用了明显重要的外部来源

### 7.4 数量约束

普通网友评论仍应设置软上限，推荐目标为不超过 20 条。

但以下内容不计入该上限：

* 作者本人内容
* 强制保留的大 V / 高影响力人物评论

如果话题极其密集，Agent 可以超过 20 条，但应以保持信息密度为前提，避免 Markdown 退化为噪音堆积。

---

## 8. 引用内容抓取原则

### 8.1 抓取范围

凡是作者 thread 或关键评论中出现的以下内容，原则上都应尽量抓取并纳入补充上下文：

* 引用推文
* 文章页面
* 博客
* 新闻链接
* 论文或文档页面
* 其他可支撑核心论点的外部网页

### 8.2 抓取深度

对引用内容的目标不是无限深挖，而是拿到足以解释其价值的最小充分信息：

* 标题
* 作者或来源
* 核心观点
* 与当前讨论的关联
* 必要时摘录关键段落或页面摘要

### 8.3 优先级规则

以下引用内容优先级最高：

* 作者本人主动引用的内容
* 被多次讨论或反复提及的内容
* 明显是争议来源或论证依据的内容

---

## 9. 评论区浏览策略

### 9.1 默认策略

X 通常会把较有代表性的评论排在靠前位置，因此 Agent 默认应优先深看前 1 到 2 屏评论区。

### 9.2 继续下翻的条件

只有在以下情况出现时，才建议继续下翻：

* 仍持续出现高价值评论
* 发现明显未闭合的争议链
* 发现更多作者回复
* 发现关键人物开始参与
* 发现引用内容尚未被充分理解

### 9.3 停止下翻的条件

若连续两轮观察都没有新增高价值节点，且当前已能清楚解释该话题的主要结构，则应停止继续下翻。

## 10. 状态机设计

### 10.1 主循环

Skill 采用受控状态机循环：

1. `Observe`
抓取当前页面截图，并记录当前滚动后可见区域。

2. `Reason`
对截图做视觉理解，识别页面中的：
* 主贴
* 作者 thread
* 作者回复
* 普通评论
* 高影响力评论
* 引用卡片
* 点赞、收藏、认证等信号

3. `Decide`
基于当前认知状态，决定下一步：
* 提取当前可见内容
* 点击引用内容下钻
* 回退到主时间线
* 继续滚动
* 停止任务

4. `Act`
执行点击、滚动、等待、回退等拟人化操作。

### 10.2 子状态

主循环之下建议细分以下子状态：

* `LOAD_ROOT`
打开目标推文并确认页面已稳定。

* `COLLECT_AUTHOR_THREAD`
优先收集作者 thread 和作者补充。

* `COLLECT_PRIORITY_REPLIES`
收集高价值评论和关键人物评论。

* `DRILL_QUOTE`
进入引用推文或引用页面，抓取上下文。

* `RETURN_MAIN`
返回主时间线并确认上下文未丢失。

* `SCROLL_DISCOVERY`
向下探索新的候选内容。

* `ASSEMBLE_OUTPUT`
整理去重结果并输出文件。

* `ABORT_OR_RECOVER`
处理异常、重试、熔断或恢复。

---

## 11. 自主结束条件

### 11.1 认知闭环标准

当 Agent 基本能够回答以下问题时，可认为任务接近完成：

* 这条推文的主题是什么
* 作者的主张和意图是什么
* 作者是否追加了关键背景
* 评论区最重要的支持与反对意见分别是什么
* 哪些关键人物参与了讨论
* 重要的引用内容有哪些
* 当前争议链是否已大致闭合

### 11.2 推荐结束信号

满足以下大部分条件即可结束：

* 作者主贴与 thread 已基本完整
* 作者在评论区的重要互动已覆盖
* 大 V / 高影响力评论已覆盖
* 关键引用内容已抓取
* 最近两轮观察无明显新增高价值节点
* 当前已足以产出高质量 Markdown

### 11.3 熔断条件

任务应具备全局熔断能力，例如：

* 最大截图次数
* 最大连续无效滚动次数
* 最大连续点击失败次数
* 最大连续页面未响应次数

一旦触发熔断，Agent 应优先保证已有数据可输出，而不是直接丢弃任务。

---

## 12. 拟人化执行控制

### 12.1 禁止项

绝对禁用：

* `window.scrollTo`
* JS 注入驱动页面行为
* 非拟人化高速连续点击

### 12.2 行为拟真

连续操作之间应加入人类行为特征：

* 随机停顿
* 小幅节奏变化
* 必要时微调滚动
* 点击失败后的轻微位置修正

### 12.3 推荐动作

优先使用以下可解释动作：

* 鼠标点击
* 滚轮滚动
* `ArrowDown`
* 浏览器回退快捷键

---

## 13. 中间数据与任务目录

### 13.1 任务级工作目录

每次抓取任务建议生成独立工作目录，例如：

```text
runs/<tweet_id>_<timestamp>/
```

### 13.2 推荐目录结构

```text
runs/<tweet_id>_<timestamp>/
  meta.json
  final.md
  observations.jsonl
  entities.json
  screenshots/
  quotes/
  links/
  logs/
```

### 13.3 文件职责

* `meta.json`
记录原始 URL、触发账号、启动时间、结束时间、任务状态等元信息。

* `observations.jsonl`
记录每次观察后的结构化摘要，例如当前可见节点、候选评论、下一步决策理由。

* `entities.json`
记录去重后的推文、评论、作者回复、引用来源等核心实体。

* `screenshots/`
保存关键截图，便于回放与调试。

* `quotes/` 与 `links/`
保存引用内容的抓取摘要、页面信息和必要截图。

* `logs/`
保存异常记录、重试记录、熔断信息等。

## 14. 去重与合并原则

### 14.1 去重目标

由于滚动截图存在重叠区域，Skill 必须具备中间态去重能力，避免：

* 同一评论重复保留
* 同一作者 thread 片段被重复拼接
* 引用内容重复下钻

### 14.2 去重依据

可综合使用以下信息进行去重：

* 文本相似度
* 作者名与 Handle
* 相对时间
* 视觉位置关系
* 点赞数等辅助信号

### 14.3 合并策略

若同一内容在不同观察轮次中信息更完整，则以后者覆盖前者，或合并生成更高置信度版本。

---

## 15. 异常处理与恢复原则

### 15.1 总体原则

失败恢复由 Agent 自主处理，但恢复目标应始终一致：

* 优先保留已有成果
* 尽量回到主任务路径
* 避免陷入死循环

### 15.2 常见异常

包括但不限于：

* 页面加载缓慢
* 点击引用卡片未跳转
* 页面元素漂移
* 内容遮挡
* 登录态异常
* 滚动后未出现有效新内容

### 15.3 恢复动作

Agent 可自主组合以下动作：

* 等待
* 小幅滚动后重试
* 重新定位点击区域
* 浏览器回退
* 页面刷新
* 跳过低优先级目标并继续主线

### 15.4 恢复上限

对单个异常目标应设置有限次重试。超过阈值后，应记录失败并继续任务，而不是卡死。

## 16. 最终 Markdown 规范

### 16.1 基本元数据

文档头部至少包含：

* 任务执行时间
* 原始 URL
* 触发账号 Handle
* 任务摘要

### 16.2 正文结构建议

建议 `final.md` 使用以下结构：

1. 任务概览
2. Root Tweet 摘要
3. 作者 Thread 全文或结构化整理
4. 作者在评论区的关键补充
5. 高价值评论精选
6. 高影响力人物评论
7. 引用推文 / 外部链接补充上下文
8. 关键争议点与观察总结

### 16.3 表达规范

应使用 Markdown 引用和层级来表达线程关系，并显式标注价值信号，例如：

* `[📌 作者回复]`
* `[🏅 高影响力账号]`
* `[👍 高赞]`
* `[🔖 高收藏]`
* `[🔗 引用来源]`

### 16.4 输出目标

最终文档应让后续模型可以直接进入分析，而不需要再次反问“背景是什么”。

---

## 17. 实现建议

### 17.1 实现重心

真正的实现重点不在浏览器控制，而在以下能力：

* 视觉解析后的结构化抽取
* 自主价值判断
* 任务级缓存和溯源
* 去重与合并
* 稳健的结束判断

### 17.2 推荐开发顺序

推荐按以下顺序实现：

1. 单条 Root Tweet 打开与截图观察
2. 作者 thread 抽取
3. 评论区高价值节点识别
4. 引用内容下钻与回退
5. 中间态存储与去重
6. Markdown 组装
7. 异常恢复与熔断

---

## 18. 结论

本文档明确规定：该 Skill 应被实现为一个拥有自主判断能力的 X 研究型 Agent。

它的关键能力不是“抓得更多”，而是：

* 像研究员一样理解内容
* 像编辑一样筛选内容
* 像分析师一样保留关键脉络
* 像工程系统一样留下可复盘的中间证据

最终目标是沉淀一份高密度、高可信、可追溯、可直接供大模型分析的 Markdown 上下文包。
