
# 产品需求文档 (PRD)：X (Twitter) 深度视觉抓取 Skill (Agentic Workflow)

## 1. 项目概述与核心愿景

### 1.1 业务动机与核心痛点
X (Twitter) 平台沉淀了大量实时的、极具价值的“思维碰撞”、“需求辩论”和“矛盾对立讨论”。这些高密度、动态演进的上下文信息是传统搜索引擎无法获取的信息盲区。
本项目旨在通过 Agentic Workflow 构建一套高阶信息获取技能（Skill），打通从“X 平台实时辩论场”到“大模型深度分析引擎”的数据管道，解决高质量实时语料获取的痛点。

### 1.2 核心业务目标
开发一个基于视觉与物理模拟的 Agent Skill。
* **明确输入**：一个目标 X 推文的分享链接。
* **明确输出**：一份结构清晰、上下文完备的 Markdown 文件。
* **交付期望**：该 Markdown 必须能够让 ChatGPT Pro 接收后，立刻“秒懂”该话题的完整来龙去脉，从而直接切入深度分析、提炼前瞻性洞察，无需再次索要背景信息。

### 1.3 最高优先级原则 (First Principles)
1. **极致安全性（账号零风险）**：视 X 账号资产安全为第一生命线。坚持“纯视觉模拟人”的第一性原理，坚决不使用任何 API、DOM 解析或网络请求拦截。通过消耗大量 Token，用“看截图、按键盘”的拟人化物理交互来获取信息，彻底规避风控。
2. **上下文的绝对完整性**：对于有价值的信息链条，必须追根溯源。若原文或评论中包含**“引用推文/文章 (Quoted content)”**，Agent 必须模拟点击进入，将其完整内容爬取下来并拼接到上下文中。

## 2. 系统架构设计：Skill 的生态位

Skill 本身不负责底层通信，而是作为一套注入到 Agent 宿主（如 Claude Code、OpenClaw）中的“标准操作程序 (SOP)”。

### 2.1 宿主与依赖环境
* **运行环境**：macOS 本地环境。
* **浏览器实例**：基于本地 User Data Directory 启动的 Chrome，承载已登录的 X 账号 Cookie，规避未登录状态的访问限制。
* **底层工具链 (MCP Server)**：提供基础的 `capture_screenshot()`, `dispatch_key_event()`, `dispatch_mouse_event()` 等原子化视觉与交互工具。

### 2.2 核心控制流：视觉认知状态机 (Visual Cognitive State Machine)
Skill 的核心是一个受控的循环状态机，包含四个关键节拍：
1. **感知 (Observe)**：调用 MCP 工具获取当前浏览器视口的无损截图。
2. **推理 (Reason)**：调用多模态模型进行视觉反向渲染。识别推文层级（Root / 作者跟帖 / 网友回复 / 引用卡片），并提取点赞数、收藏数、身份认证等价值指标。
3. **决策 (Decide)**：基于推理结果更新状态转移条件（如下钻引用页面、向下滚动、或触达底部结束抓取）。
4. **行动 (Act)**：根据决策，调用 MCP 执行具体的物理动作（如坐标点击、发送模拟按键）。

## 3. 技能功能需求详细说明

面对复杂且存在大量噪音的 X 评论区，Agent 在执行视觉解析与暂存区聚合时，必须严格执行以下数据筛选规则：

### 3.1 必须抓取的基石数据 (Must Have)
* **时间线完整性**：原作者发布的完整 Thread（多条连续推文）必须一字不落拉取。
* **作者交互**：原作者在评论区所做的所有额外补充说明、以及作者回复他人的互动帖，必须全量保留。这是理解作者核心意图的关键。
* **引用溯源**：所有包含实质性上下文的被引用文章/推文，必须计算屏幕坐标并模拟点击下钻，抓取完整原文后，再模拟快捷键（`Cmd + Left Arrow`）回退到主时间线。

### 3.2 网友评论的严选规则 (High Priority & Constraints)
为确保最终输出的 Markdown 文件信息密度极高，针对网友评论设定以下硬性约束：
1. **价值导向**：重点捕捉“高赞 (High Likes)”和“高收藏 (High Bookmarks)”的优质回复。
2. **意见领袖优先**：视觉 OCR 需识别用户身份标记，若回复者为官方认证用户、大V或行业名人，其评论应被优先保留。
3. **数量熔断阈值 (Hard Limit)**：无论该帖有多火爆，最终输出的 Markdown 文件中，**清洗后保留的网友评论总数绝对不得超过 20 条**（注：此 20 条限制仅针对网友，不包含作者本人的所有回复）。

### 3.3 拟人化执行控制引擎
* **绝对禁用**任何形式的 `window.scrollTo` 或 JavaScript 注入操作。
* **动态噪音注入 (Fuzzing)**：在连续的滚动或点击操作之间，必须引入具有人类行为特征的随机停顿（如 1500ms - 4200ms）。

---

## 4. 输出规范 (Markdown Deliverable)

当状态机判定抓取生命周期结束时，执行最终的清洗和组装，生成用于 ChatGPT Pro 深度分析的 Markdown 文件。

**文件元数据及格式要求：**
* 头部包含基本信息：任务执行时间（采用 24 小时制，例如 `14:35:00`）、原始 URL、触发账号 Handle。
* 层级关系表达：使用 Markdown 的引用语法 `>` 及缩进，清晰表达推文的树状拓扑结构。
* 视觉锚点：显性标注价值指标（如 `[👍 12.5k]`, `[🔖 3.2k]`, `[📌 作者回复]`, `[🏅 认证大V]`），以便分析大模型快速锁定核心争议点。

---

## 5. 异常处理与降级策略

* **暂存区管理与滑动去重**：由于拟人化滚动必然导致前后两张截图存在内容重叠，Skill 内部需实现中间态数据的本地化管理，并具备基于文本相似度的去重能力，确保拼凑出来的 Thread 逻辑连贯。
* **视觉漂移补偿**：若尝试点击某引用推文但页面未发生预期跳转，Agent 需主动下发小幅度的 `ArrowDown` 滚动以调整元素位置，重新计算坐标并重试。
* **超时与死循环熔断**：设定全局最大截图次数阈值，防止因网络卡顿或极端 UI 布局导致的无限循环。
